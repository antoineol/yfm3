# P6.5 Refactor

## Current State

- DONE: the suggestion flow now uses one direct path per concern.
- Compute: rank the unique one-for-one swaps by sampled delta, then exact-score a small shortlist in a worker.
- Show: render only loading or the single best positive swap, with apply/reject actions.
- Apply: inline the Convex legality checks in `applySuggestedSwap` and perform a plain delete+insert swap.

## Features To Keep

- Suggest a swap only for the last added card.
- Run only when the deck is full.
- Consider only one-for-one swaps.
- Pick the single best positive swap from a small exact-scored shortlist.
- Show nothing if no swap strictly improves score.
- Let the user apply or reject the suggestion.
- On apply, the server checks:
  - the removed card is still in the deck
  - the added card still has an available owned copy
  - optionally, `addCardId !== removeCardId`
- Hide the suggestion after a successful apply.
- Use a web worker so scoring does not freeze the UI.
- Show loading while computing.
- Show the score improvement in the UI.
- Show an error toast if apply fails.
- Recompute when deck inputs really change, but avoid useless recomputations from reference-only Convex churn.
- Keep the shortlist small enough to stay sub-second on a full 40-card deck.

## Features To Drop

- Extra helper layers for trivial validation.
- Extra returned fields not needed by the UI.
- Tie-break rules that are not user-visible.
- Preserve deck order when applying the swap.
- Revalidating target deck size when applying the swap.
- Fetching preferences on apply only to enforce deck size.
- Exact-scanning every possible swap if that pushes latency back into multi-second territory.

## Refactor Instructions

- Start from the kept features only. Everything else is expendable.
- Prefer one direct code path per concern: compute suggestion, show suggestion, apply suggestion.
- Inline simple checks instead of adding validation wrappers or helper files.
- Prefer the smallest ranked-then-exact shortlist that keeps the worker comfortably under one second.
- Keep data flow obvious. Avoid passing through extra fields that the caller does not need.
- Remove abstractions that do not simplify reasoning.
- Remove indirections added only for architecture purity.
- Keep the code compact, but not cryptic.
- Keep tests focused on user-visible behavior and core legality checks.

## Next Step After This

- Do not add complexity back into this path unless measurement justifies it.
- If latency becomes a real problem again, optimize the shortlist size or ranking quality before reconsidering a full exact scan.
