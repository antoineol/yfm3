# P1.1: Authentication (Clerk + Google OAuth) — DONE

**Priority:** P0

## Why

The app must use real cross-device identity. Manual `userId` input was a dev artifact and allowed trivial impersonation.

## Final Decision

- Provider: Clerk
- Sign-in method: Google only
- Frontend stack: React + Vite
- Backend stack: Convex with Clerk JWT validation

## Current State

- The frontend uses Clerk via `@clerk/clerk-react`
- Convex uses Clerk auth config in [`convex/auth.config.ts`](/home/ubuntu/perso/yfm3/convex/auth.config.ts)
- Backend functions read the authenticated identity through [`requireAuth`](/home/ubuntu/perso/yfm3/convex/authHelper.ts)
- App data stays keyed by `userId: v.string()`, now storing the Clerk-backed Convex token identifier
- Legacy auth files and auth tables were removed

## Required Configuration

### Local frontend env

Set in `.env.local`:

```bash
VITE_CLERK_PUBLISHABLE_KEY=pk_...
VITE_CONVEX_URL=https://<deployment>.convex.cloud
```

Use [`.env.example`](/home/ubuntu/perso/yfm3/.env.example) as the local template.

Do not add `CLERK_FRONTEND_API_URL` to the Vite app config unless you have a separate reason to keep it locally. The frontend does not use it.

### Convex deployment env

Set on the Convex deployment:

```bash
CLERK_FRONTEND_API_URL=https://<your-instance>.clerk.accounts.dev
```

### Clerk dashboard

Required Clerk setup:

1. Create a React app in Clerk.
2. Enable Google auth.
3. Create a JWT template named exactly `convex`.
4. Use the same Clerk instance values in local env and Convex env.

Without the `convex` JWT template, Clerk sign-in can succeed while Convex remains unauthenticated.

## Data Migration

Old rows keyed by previous dev or legacy auth identities are not migrated automatically.

Accepted tradeoff:
- Existing local/dev data may become orphaned
- Re-entering a collection is acceptable for this project

If needed later, a one-off Convex mutation can reassign rows from an old `userId` to a new one.

## Verification

Expected auth flow:

1. User clicks `Sign in with Google`
2. Clerk completes Google auth
3. Clerk mints the `convex` JWT template
4. Convex accepts the token
5. `useConvexAuth()` becomes authenticated
6. App loads user-scoped data normally
